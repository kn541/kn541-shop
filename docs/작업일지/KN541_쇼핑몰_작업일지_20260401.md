## 3. 아키텍처 설계

```
[Next.js 페이지]
    ↓ 호출
[data/data.ts] ← 기존 컴포넌트가 그대로 호출하는 인터페이스 유지
    ↓ API 우선
[lib/api-products.ts] → [lib/api.ts] → KN541 Railway FastAPI
    ↓ 실패 시 폴백
[더미데이터 (기존 정적 데이터)]
    ↓ 성공 시 변환
[lib/adapters.ts] → TProductItem / TCollection 형식
```

### 핵심 설계 원칙
- **컴포넌트 무수정**: 기존 Ciseco 컴포넌트는 코드 변경 없이 그대로 사용
- **폴백 자동화**: NEXT_PUBLIC_API_URL 미설정 또는 API 오류 시 더미데이터로 자동 전환
- **점진적 전환**: 환경변수 하나로 더미 ↔ 실제 데이터 전환 가능

## 4. API 연동 규칙 (KN541 표준)

| 항목 | 규칙 |
|------|------|
| 환경변수 | `NEXT_PUBLIC_API_URL` (Vercel 환경변수에 Railway URL 입력) |
| 응답 형식 | `{ status: "success", data: { items: [...], total: N } }` |
| 인증 | 공개 API는 토큰 불필요. 마이페이지/주문 등은 `Bearer {access_token}` |
| 캐시 | 상품/카테고리: revalidate 60초. 인증 필요 데이터: no-store |
| 하드코딩 금지 | 코드값은 `/global-variables?category=xxx` API로 동적 로드 |

## 5. 백엔드 확인 필요 사항

현재 연동 중인 엔드포인트 목록. 백엔드 창에서 경로 확인 후 `src/lib/api-products.ts` 수정 필요:

| Method | 경로 | 용도 | 확인 |
|--------|------|------|------|
| GET | `/products` | 상품 목록 (page, size, category, sort, q 파라미터) | ⏳ 확인 필요 |
| GET | `/products/{handle}` | 상품 단건 조회 | ⏳ 확인 필요 |
| GET | `/products/categories` | 카테고리 목록 | ⏳ 확인 필요 |
| GET | `/products/categories/{handle}` | 카테고리 단건 조회 | ⏳ 확인 필요 |

> ⚠️ 백엔드 API 경로가 다를 경우 `src/lib/api-products.ts`의 경로만 수정하면 됩니다.
